# 2D 턴제 디펜스 게임 - 아키텍처 & 개발 방법론 멘탈 모델

## 1. 장르 특성 분석

### 2D 턴제 디펜스 게임이란?
플레이어가 **방어 측(마왕/건축가)**이 되어 던전을 설계하고, **공격 측(용사/침입자)**의 침입을 턴 기반으로 막아내는 게임.

### 핵심 게임 루프
```
┌─────────────────────────────────────────────────┐
│                  코어 게임 루프                    │
│                                                   │
│   [준비 페이즈]  →  [실행 페이즈]  →  [정산 페이즈]  │
│       │                 │                │        │
│   방 배치           용사 탐험         보상/결과     │
│   몬스터 배치       전투 발생         재화 획득     │
│   자원 관리         AI 행동           진행도 갱신   │
│       │                 │                │        │
│       └────────────────←┘────────────────┘        │
│                    (반복)                          │
└─────────────────────────────────────────────────┘
```

### 장르 고유 요구사항
| 요구사항 | 설명 | 아키텍처 영향 |
|---------|------|-------------|
| 턴 관리 | 명확한 턴/페이즈 전환 | State Machine 필수 |
| 그리드 시스템 | 타일 기반 배치 | 데이터-뷰 분리 |
| 전투 시뮬레이션 | 스킬, 스탯, 타겟팅 | Command Pattern |
| AI 의사결정 | 적 유닛의 자율 행동 | Strategy Pattern |
| 밸런싱 | 수치 조정이 빈번 | Data-Driven Design |
| 진행 시스템 | 스테이지, 언락, 성장 | 영속성 시스템 |

## 2. 아키텍처 멘탈 모델

### 2.1 핵심 원칙: "데이터와 로직의 분리"

```
┌──────────────────────────────────────────────┐
│              아키텍처 3계층 모델               │
│                                              │
│  ┌────────────────────────────────────────┐  │
│  │         프레젠테이션 계층 (View)         │  │
│  │   UI, 애니메이션, VFX, 사운드           │  │
│  │   "보여주는 것만 담당"                   │  │
│  └──────────────┬─────────────────────────┘  │
│                 │ 이벤트 구독                  │
│  ┌──────────────▼─────────────────────────┐  │
│  │         게임 로직 계층 (Logic)           │  │
│  │   턴 관리, 전투, AI, 경제 시스템          │  │
│  │   "규칙과 흐름을 담당"                   │  │
│  └──────────────┬─────────────────────────┘  │
│                 │ 데이터 참조                  │
│  ┌──────────────▼─────────────────────────┐  │
│  │          데이터 계층 (Data)              │  │
│  │   ScriptableObject, 저장 데이터          │  │
│  │   "수치와 상태만 담당"                   │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

**왜 이 구조인가:**
- 밸런싱 작업 시 데이터만 수정하면 되므로 코드 변경 불필요
- UI 변경이 게임 로직에 영향을 주지 않음
- 각 계층을 독립적으로 테스트 가능

### 2.2 핵심 디자인 패턴 6가지

#### Pattern 1: State Machine (상태 머신)
**용도:** 게임 페이즈, 전투 턴, 엔티티 상태 관리

```
[GameState]
  Lobby → Preparation → Exploration → GameOver
                            │
                        [BattleState]
                  Setup → PlayerTurn → EnemyTurn → Result
```

**적용 규칙:**
- 게임 전체 흐름 = GameStateMachine
- 전투 흐름 = BattleStateMachine
- 개별 유닛 행동 = EntityStateMachine
- 각 State는 `Enter()`, `Execute()`, `Exit()` 인터페이스 구현

**왜 이 패턴인가:** 턴제 게임은 본질적으로 "지금 누구의 차례인가"를 추적하는 상태 머신이다. 중첩 if문 대신 상태 클래스로 분리하면 새로운 페이즈 추가가 용이하다.

#### Pattern 2: Command Pattern (명령 패턴)
**용도:** 모든 게임 액션(공격, 이동, 스킬 사용)을 객체화

```csharp
// 멘탈 모델
ICommand {
    Execute()
    Undo()      // 되돌리기 지원
    GetPreview() // 미리보기 지원
}

// 사용 예시: 전투 액션
AttackCommand → Execute() → 데미지 계산 → 결과 이벤트 발행
MoveCommand   → Execute() → 위치 변경 → 이동 이벤트 발행
SkillCommand  → Execute() → 효과 적용 → 스킬 이벤트 발행
```

**왜 이 패턴인가:**
- 플레이어 입력과 AI 모두 동일한 Command로 행동 → 코드 중복 제거
- Undo/Redo 기능 구현 가능 (턴제에서 "되돌리기" 자연스러움)
- 전투 리플레이 시스템 구현 가능
- 네트워크 멀티플레이 확장 시 Command를 직렬화하여 전송

#### Pattern 3: Event-Driven Architecture (이벤트 기반 아키텍처)
**용도:** 시스템 간 느슨한 결합

```
[전투 시스템] ─── OnDamageDealt 이벤트 발행 ───→ [UI] 데미지 표시
                                              ├→ [VFX] 이펙트 재생
                                              ├→ [Audio] 타격음 재생
                                              └→ [통계] 데이터 기록
```

**Unity 구현 방식 비교:**

| 방식 | 장점 | 단점 | 추천 상황 |
|------|------|------|----------|
| C# delegate/event | 성능 최고, 타입 안전 | 참조 관리 필요 | 동일 시스템 내 |
| ScriptableObject 이벤트 채널 | Inspector에서 설정, 디자이너 친화적 | 약간의 오버헤드 | 시스템 간 통신 |
| UnityEvent | Inspector 설정 가능 | 성능 낮음, 디버깅 어려움 | 단순 UI 바인딩 |

**추천:** ScriptableObject 이벤트 채널을 기본으로 사용하되, 고빈도 호출에는 C# event 사용.

#### Pattern 4: Data-Driven Design (데이터 주도 설계)
**용도:** 게임 콘텐츠(몬스터, 스킬, 방 타입)의 확장성

```
ScriptableObject 계층 구조:

EntityData (기본 스탯)
  ├── MonsterData (몬스터 고유 속성)
  ├── ChampionData (용사 고유 속성)
  └── BossData (보스 고유 속성)

SkillData (스킬 정의)
  ├── 데미지 스킬
  ├── 힐 스킬
  └── 버프/디버프 스킬

RoomData (방 타입 정의)
  ├── 전투방
  ├── 함정방
  └── 특수방
```

**핵심 규칙:**
- 새로운 몬스터/스킬/방 추가 = ScriptableObject 에셋 생성만으로 완료
- 코드 수정 없이 밸런싱 가능
- 조합 폭발 대응: 개별 효과를 작은 단위로 분리 후 조합

#### Pattern 5: Strategy Pattern (전략 패턴)
**용도:** AI 행동, 타겟팅 로직, 경로 탐색 알고리즘

```
IPathfindingStrategy
  ├── BFSPathfinding     (기본 용사)
  ├── AStarPathfinding   (똑똑한 용사)
  └── RandomPathfinding  (방황하는 용사)

ITargetingStrategy
  ├── ClosestTarget      (가장 가까운 적)
  ├── WeakestTarget      (가장 약한 적)
  └── HighestThreat      (가장 위협적인 적)
```

**왜 이 패턴인가:** 용사마다 다른 AI 행동을 부여할 때, if-else 분기 대신 전략 객체를 교체하면 새로운 AI 추가가 독립적이다.

#### Pattern 6: Object Pool (오브젝트 풀)
**용도:** 빈번한 생성/파괴 객체 관리 (이펙트, 데미지 텍스트, 투사체)

**적용 대상:**
- 전투 이펙트 (VFX)
- 데미지 팝업 텍스트
- 적 유닛 (웨이브 스폰 시)
- UI 요소 (인벤토리 아이템 등)

### 시스템 간 통신 규칙
1. **수직 통신:** 매니저 → 하위 시스템 (직접 호출)
2. **수평 통신:** 시스템 → 시스템 (이벤트 채널을 통해서만)
3. **UI 통신:** 이벤트 구독만 (UI는 절대 게임 로직을 직접 호출하지 않음)
4. **데이터 접근:** ScriptableObject 참조 (런타임 데이터는 별도 관리)

## 4. 개발 방법론 멘탈 모델

### 4.1 추천 방법론: "스프린트 기반 반복적 프로토타이핑"

기존 Agile/Scrum을 게임 개발에 맞게 변형한 방법론.

```
┌─────────────────────────────────────────────────┐
│          스프린트 기반 반복적 프로토타이핑          │
│                                                   │
│   ┌─────────┐   ┌─────────┐   ┌─────────┐       │
│   │ 스프린트 │ → │ 스프린트 │ → │ 스프린트 │ → ... │
│   │  1주     │   │  1주     │   │  1주     │       │
│   └────┬────┘   └────┬────┘   └────┬────┘       │
│        │              │              │             │
│    플레이테스트     플레이테스트     플레이테스트     │
│    + 회고          + 회고          + 회고          │
└─────────────────────────────────────────────────┘
```

### 4.2 스프린트 구조 (1주 단위)

| 일차 | 활동 | 산출물 |
|------|------|--------|
| Day 1 | 스프린트 계획 + 설계 | 기능 명세, 태스크 분해 |
| Day 2-4 | 구현 | 동작하는 기능 |
| Day 5 | 플레이테스트 + 밸런싱 | 피드백, 수치 조정 |
| Day 6 | 버그 수정 + 폴리싱 | 안정화된 빌드 |
| Day 7 | 회고 + 다음 계획 | 개선점, 다음 목표 |

### 4.3 개발 순서 원칙: "핵심부터 껍데기까지"

```
Phase 1: 코어 메카닉 (재미 검증)
  └→ 그리드 배치 + 기본 전투 = "이 게임이 재미있는가?"

Phase 2: 게임 루프 (흐름 검증)
  └→ 준비 → 실행 → 정산 사이클 = "반복 플레이가 되는가?"

Phase 3: 콘텐츠 확장 (깊이 검증)
  └→ 다양한 몬스터, 스킬, 방 타입 = "다양성이 있는가?"

Phase 4: 메타 시스템 (성장 검증)
  └→ 영구 업그레이드, 언락 = "계속 하고 싶은가?"

Phase 5: 폴리싱 (완성도)
  └→ UI, VFX, 사운드, 밸런싱 = "출시 가능한가?"
```

### 4.4 "재미 우선" 원칙

```
의사결정 프레임워크:

"이 기능이 게임을 더 재미있게 만드는가?"
    ├── YES → 구현 우선순위 ↑
    ├── NO  → 보류 또는 삭제
    └── 모르겠음 → 최소 프로토타입으로 테스트
```

**안티패턴 경고:**
- 재미 검증 전에 완벽한 아키텍처를 추구하지 않기
- "나중에 필요할 것 같아서" 미리 구현하지 않기 (YAGNI)
- 밸런싱은 콘텐츠가 어느 정도 갖춰진 후에 시작

## 5. 현재 프로젝트(던전 메이커) 적용 분석

### 5.1 잘 적용된 부분
| 현재 구현 | 패턴 | 평가 |
|----------|------|------|
| GameStateManager (Preparation/Exploration) | State Machine | 기본 구조 양호 |
| ScriptableObject (MonsterData, SkillData) | Data-Driven | 확장성 좋음 |
| 델리게이트 이벤트 (OnGoldChanged 등) | Event-Driven | 느슨한 결합 달성 |
| GridManager + RoomManager | 관심사 분리 | 역할 분리 적절 |

### 5.2 개선 가능한 부분
| 현재 | 개선 방향 | 효과 |
|------|----------|------|
| 싱글톤 남용 가능성 | 이벤트 채널로 대체 가능한 부분 식별 | 테스트 용이성 향상 |
| BattleManager 내 전투 로직 집중 | Command Pattern으로 액션 분리 | Undo/Replay 가능 |
| 용사 AI 단일 전략 | Strategy Pattern으로 AI 다양화 | 콘텐츠 확장성 |
| PlayerPrefs 저장 | 직렬화 시스템 고도화 고려 | 복잡한 세이브 대응 |

## 6. 의사결정 가이드

### "이 기능을 어떻게 구현할까?" 플로우차트

```
새 기능 요구사항
    │
    ├── 데이터인가? (수치, 속성, 설정)
    │       └── YES → ScriptableObject로 정의
    │
    ├── 행동인가? (공격, 이동, 스킬)
    │       └── YES → Command Pattern으로 구현
    │
    ├── 상태 전환인가? (페이즈, 턴, 모드)
    │       └── YES → State Machine으로 관리
    │
    ├── 여러 시스템이 반응해야 하는가?
    │       └── YES → 이벤트 채널로 통신
    │
    ├── AI 행동의 변형인가?
    │       └── YES → Strategy Pattern으로 교체 가능하게
    │
    └── 빈번한 생성/파괴가 있는가?
            └── YES → Object Pool 사용
```

│   └── Placement/
├── AI/                 # AI 시스템
│   ├── Strategies/     # AI 전략

## 8. 요약 - 한 장으로 보는 멘탈 모델

```
┌─────────────────────────────────────────────────────┐
│          2D 턴제 디펜스 게임 멘탈 모델 요약           │
├─────────────────────────────────────────────────────┤
│                                                     │
│  [아키텍처]                                          │
│   3계층: Data → Logic → View                        │
│   6패턴: StateMachine, Command, Event,              │
│          DataDriven, Strategy, ObjectPool            │
│                                                     │
│  [방법론]                                            │
│   1주 스프린트 × 반복적 프로토타이핑                   │
│   재미 우선 → 핵심 메카닉 → 루프 → 콘텐츠 → 폴리싱    │
│                                                     │
│  [원칙]                                              │
│   1. 데이터와 로직을 분리하라                         │
│   2. 이벤트로 시스템을 연결하라                        │
│   3. 행동은 Command로 객체화하라                      │
│   4. 상태 전환은 State Machine으로 관리하라            │
│   5. 재미 검증 없이 아키텍처에 투자하지 마라            │
│   6. YAGNI - 지금 필요한 것만 구현하라                 │
│                                                     │
└─────────────────────────────────────────────────────┘
```
